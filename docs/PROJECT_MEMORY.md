# AI_gf_momo 项目级记忆

本文件记录项目已经确定的架构结论、已经废弃的方案和后续修改边界。它不是角色记忆，也不是聊天记录。修改对话、状态、生图链路前，先阅读本文件和 `docs/ARCHITECTURE_INDEX.md`。

## 当前架构基线

一轮正常对话遵循：

```text
用户输入 -> MomoAgent -> VisualContinuityAgent -> 提交状态 -> 冻结快照并创建 ImageJob -> 回复与后台生图
```

- MomoAgent 只负责角色扮演、剧情推进、自然回复和高层图片目标，不输出状态操作、服饰投影、场景标签或最终提示词。
- VisualContinuityAgent 负责视觉状态和最终画面计划，输入当前消息、角色实际回复、最近最多 8 轮视觉相关对话、上一轮状态和完整角色视觉锚点。
- 状态必须先提交，图片任务必须携带提交时冻结的快照，后台不能重新读取最新状态覆盖本轮事实。
- `state_snapshot.json` 是结构化状态源，`status.md` 是给主 Agent 和前端看的客观事实投影，两者必须同步。

## 状态与服饰不变量

- 服饰槽位固定为 `upper`、`lower`、`legwear`、`footwear`、`accessories`；Bra 和内裤分别属于 `upper/lower` 的内层。
- 每件服饰从状态开始就使用一个完整短语，例如 `white_lace_panties`，禁止重新拆成 `white`、`lace`、`panties`。
- `topless`、`bottomless`、`nude`、`completely_nude` 是裸露事实，不能因为局部特写而省略；当前实际服饰也不能被导演随意删除。
- `legwear` 与 `footwear` 分离；完全裸体时用 `completely_nude` 表达整体事实，不重复堆叠冗余裸露标签。
- 新角色和清空记录后的角色使用显式 `initialized=false`，不能把“未构建”解释成裸体、空场景或默认服饰。
- 心情不是持久化状态，不写入 `status.md`、`state_snapshot.json` 或前端状态栏。

## 提示词基线

- 最终顺序为：质量词和 rating → 角色标签/体型/外貌 → 当前服饰和裸露事实 → 构图视角 → 姿势动作 → 环境和光线。
- 角色锚点与服饰事实优先保持；导演对动作、场景、光线和自然语言进行取舍。
- 景别和部位焦点二选一；角度可以独立保留。
- 动作和环境可以用简短自然语言补充道具关系和空间关系，但不能写成小作文。
- 约 40 个语义单元只是导演的编辑目标，不是后端硬上限；长度不能取消图片或整轮对话。后端只做轻量的动作/环境短语整理。
- 禁止数值权重、权重括号和“超过数量就拒绝生图”的规则。

## 已废弃、禁止重新引入的方案

- MomoAgent 主链路不再负责 `state_ops`、`effects` 或状态变化。
- `StateReducer` 不作为正常运行时主链路；状态变化由 VisualContinuityAgent 理解并提交。
- 不恢复“状态没变化就不生图”的判断。只要 MomoAgent 给出有效 `image_goal`，就应创建图片任务。
- 不恢复“视觉同步失败就追加 state-only 模型调用”的恢复链。导演自身最多修复一次；仍失败时只能使用上一轮冻结状态做本地基础降级，不再追加模型请求。
- 不恢复 25 标签硬限制、提示词超长拦截、自动拒绝整轮回复或局部特写强行删除服饰事实。
- 不把现实日期、系统时钟或聊天技术时间注入故事、状态、主 Agent 或 MemoryAgent。
- 不修改角色 `identity.md`，除非用户明确要求。

## 场景与初始场景

- 场景切换支持自动和手动两种模式，使用独立 `scene_transition` 消息，但仍走 MomoAgent → VisualContinuityAgent → 状态提交事务。
- 初始场景保存在 `profile.json.initial_scene`，编辑模板不改变当前剧情；清空记录后按模板重新构建事实，但不要求复刻原措辞、动作或旁白。
- 新场景成功后持久化 `scene_divider`，供前端显示分割线；内部指令不写成普通用户台词。

## 记忆边界

- 向量召回只提供本轮参考，不能直接刷新长期记忆。
- `memory_candidate` 必须由后台 MemoryAgent 审核后才能写入长期记忆。
- MemoryAgent 不接收现实日期、时钟或聊天技术时间；故事里程碑只按明确故事时间或发生先后整理。
- 长期连续性由四层共同组成：窗口内完整对话、`conversation_summary.md` 窗口外连续剧情摘要、`long_term.md` 稳定长期事实、按需向量细节召回；四者不能互相替代。
- `context.compress_at` 是后台历史压缩的提前触发阈值。达到阈值后，当前回复不等待摘要模型；MemoryAgent 在后台合并旧摘要与即将离开窗口的完整消息，摘要和压缩游标在 `memory_runtime.json` 中作为同一事务提交，再投影到 `conversation_summary.md`。
- 每个角色同一时间最多运行一个上下文压缩任务；过期任务不得覆盖已经推进的摘要。清空角色记录会同时清除摘要和游标。
- MemoryAgent 写入 `soul.md` 或 `long_term.md` 后必须立即使 MomoAgent 的系统提示缓存失效，不能依赖重启后才生效。

## 第一阶段性能优化状态

已完成：

- 一轮内复用同一份持久化聊天历史，VisualContinuityAgent 不再重复读取历史文件。
- 热路径读取聊天记录时不再执行旧历史修复和图片历史合并；迁移/修复入口仍保留给普通读取和维护流程。
- MomoAgent 支持接收已组装的本轮最终提示词，不再在运行时重复组装完整提示词。
- 用户消息和角色回复通过一次批量写入保存。
- 每日记录、聊天历史、向量写入、记忆计数和图片历史落盘均不再阻塞事件循环。
- 图片任务在低优先级记忆和向量任务之前排队提交。
- VectorStore 按角色在进程内复用；角色重置或删除前会释放对应缓存，避免复用已删除目录的 Chroma 句柄。
- 向量写入使用每角色单队列、批量落库。待落库的最近回合会合并进按需召回，避免连续交互时遗漏上一轮细节。
- 向量清理改为每累计 50 条写入执行一次，不再逐回合扫描整个库；上下文压缩和定期记忆沉淀也各自保持每角色单个在途任务。
- `settings.json`、领域路由和领域知识 Markdown 按文件修改时间缓存；编辑文件后会自动失效重读。

后续只需根据实际模型能力校准 `context.max_tokens`；完整对话窗口已按预算线性选择，窗口外历史由后台滚动摘要承接。

这些优化不应通过改变 Agent 职责或增加新的模型调用解决。

以上优化不能改变“状态先提交、图片携带冻结快照、回复与状态一致”的核心顺序。

## 修改前检查清单

- 先读本文件和 `docs/ARCHITECTURE_INDEX.md`。
- 搜索旧字段和调用点，确认不是已经废弃的兼容入口。
- 不把角色运行数据、聊天记录、图片或向量库当作代码提交。
- 修改后运行架构、状态、事务、运行时、记忆、工作流和 ComfyUI 探针。
- 检查 `git diff --check` 与 `git status`，确认没有把用户数据混入修改。
